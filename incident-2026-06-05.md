# Executive Summary — Insiden "no free connection available to host", 2026-06-05

## 1. Ringkasan Insiden

| Aspek | Keterangan |
|---|---|
| **Waktu** | 2026-06-05 ~10:25–10:46 WITA |
| **Gejala** | Aplikasi `xv2.zahironline.com` mengembalikan "no free connection available to host" ke pengguna |
| **Dampak** | Throughput cluster turun dari ~150 req/sec ke ~30 req/sec; ASG auto-scale dari 2 → 10 instance; PHP-FPM saturated 80/80 children di seluruh node |
| **Status** | **RESOLVED** pukul ~10:47 WITA. Tidak ada error baru sejak 10:46:38 |

## 2. Root Cause

Insiden disebabkan oleh **rantai sebab-akibat bertingkat**, bukan satu kegagalan tunggal:

1. **Shared Redis cache antar aplikasi** — `redis-prod.zahironline.com` db1 dipakai bersama oleh API utama, HR app (`zhr-zoiali-hr-prd`), dan ≥10 aplikasi lain di subnet 10.10.10.0/24.

2. **Endpoint destruktif tersedia di API** — `DELETE /api/v2/caches` di `Controller.php:777` memanggil `Artisan::call('cache:clear')` → `FLUSHDB` (hapus seluruh database Redis). Sengaja dibuat untuk operasional, tapi blast radius mengenai semua app yang share db1.

3. **Pola N+1 cache lookup di mapper** — endpoint berat seperti `GET /api/v2/sales_invoices?per_page=200&includes[...]` memanggil `CompanySettings::getByKey()` per row → **>3.000 GET Redis untuk key yang sama dalam 2 detik per request** (terkonfirmasi via `MONITOR`).

4. **Trigger yang memicu insiden** — beberapa FLUSHDB berturut-turut dari aplikasi lain (terutama IP-IP non-ASG seperti 10.10.10.5 = `zhr-zoiali-hr-prd`) menghapus cache. Request `sales_invoices` yang sedang in-flight gagal cache, fall back ke DB, request berikutnya antre, semua 80 php-fpm children stuck di `fgets()` Predis → Aliyun SLB melihat backend tidak respon → mengembalikan **"no free connection available to host"** ke klien.

**Bukti yang mengukuhkan diagnosis:**

- Redis SLOWLOG: **127/127 entries semua `FLUSHDB`**
- PHP-FPM slowlog: ratusan stack trace identik berakhir di `fgets()` Predis
- nginx error log: spam `recv() failed (104: Connection reset by peer)` saat insiden, hilang setelah perbaikan

## 3. Tindakan Mitigasi (Selesai)

| Aksi | Cakupan | Waktu |
|---|---|---|
| Migrasi cache aplikasi dari Redis `db1` → `db5` (terisolasi) | Runtime di 10 instance ASG + GitOps source of truth | 10:44–10:46 |
| Commit GitOps untuk persistensi ASG launch berikutnya | `zahirenv/config@main` commit `21252be` | 11:53 |
| Verifikasi pemulihan | Error rate → 0; pool 13 active / 63 idle per instance | 10:47+ |

**Sisa utang teknis** (belum disentuh, perlu rencana terpisah):

- Refactor `clearCaches()` agar tidak FLUSHDB (gunakan invalidasi key-spesifik)
- Refactor pola N+1 di mapper SalesInvoices (memoize `CompanySettings` per request)
- Rotasi PAT GitHub yang masih hard-coded di user-data ASG

## 4. Indikator Akses Postman yang Patut Dicurigai

Investigasi log API berhasil menemukan **17 hit total** ke endpoint sensitif `DELETE /api/v2/caches` dalam ~24 jam terakhir, terbagi menjadi 8 unique requests (sisanya artefak log replication antar instance).

| Sumber | Bukti | Catatan |

| **IP `27.123.223.206`** (Telkomsel — jaringan **konsumen/seluler**) | 3 hits (2026-06-05 03:46:13/17/19) | IP residensial/seluler, jam 03:46 dini hari WIB |

| **Bearer token Authorization** | `zrDZaIVCNUKphfjKpTAHA6uoVmASv8JnA2AjLGm5` — **identik di semua 13 hit dari 2 IP berbeda** | Token yang sama dipakai dari jaringan korporat **dan** jaringan seluler |

### B. Indikator Yang Membenarkan Kecurigaan

1. **Token Authorization yang sama dipakai dari 2 jaringan ISP berbeda** (Biznet Jakarta korporat + Telkomsel konsumen). Ini bisa berarti:
   - Satu orang menggunakan token saat di office, lalu di rumah/seluler, **ATAU**
   - Token diteruskan ke pihak ketiga, **ATAU**
   - Token bocor/dicuri dari workstation pemilik asal

2. **Akses dini hari (03:46 WIB)** dari jaringan seluler konsumen — bukan jam kerja normal, bukan dari office IP, untuk endpoint operasional yang nominalnya hanya dipakai untuk maintenance.

3. **Endpoint yang dipakai destruktif** (`FLUSHDB` seluruh cache) — tidak ada alasan operasional rutin untuk memanggilnya berkali-kali tanpa konteks insiden.

4. **Tidak ada `client_id` / `client_name`** di payload log — request tidak diasosiasikan dengan aplikasi resmi mana pun. Semua jalur akses lewat token raw, bukan via OAuth client legitimate.

5. **Bearer token tidak terkait identitas user spesifik di log** — field `user.email` hanya `system`, artinya tidak terlacak ke akun pegawai. Ini menyulitkan attribution dan justru menunjukkan kelemahan audit-trail.

### C. Hipotesis yang Konsisten dengan Bukti

- **Mantan anggota tim teknis** yang masih memegang token operasional yang belum dirotasi
- Token `zrDZaIVCNUKphfjKpTAHA6uoVmASv8JnA2AjLGm5` adalah **persistent Bearer token** (bukan session token) yang masih valid setelah orang yang bersangkutan meninggalkan organisasi
- Akses dari 2 ISP (korporat + konsumen) konsisten dengan pola seseorang yang **tidak lagi punya akses ke jaringan kantor** secara konsisten, sehingga menggunakan jaringan pribadi
- Aktivitas dini hari + endpoint berbahaya + token tanpa attribution + tanpa kompetensi formal yang mengirim ticket terlebih dahulu = pola yang anomali bagi pegawai aktif

### D. Yang Belum Bisa Dipastikan (Perlu Penyelidikan Lanjut)

- **Siapa pemilik asal token `zrDZaIVCNUKphfjKpTAHA6uoVmASv8JnA2AjLGm5`** — perlu cek tabel `oauth_access_tokens` / `personal_access_tokens` / tabel terkait di DB untuk lihat `user_id`/`created_by`/`name`/`created_at`
- **Apakah pemilik asal token masih karyawan aktif** — perlu konfirmasi dari HR
- **Apakah ada akses lain dengan token yang sama** ke endpoint lain — perlu scan log lebih luas (bukan hanya `DELETE /api/v2/caches`)
- **IP 101.255.138.6 dan 27.123.223.206 milik siapa** — perlu cek WHOIS dan korelasi dengan daftar IP rumah/kantor karyawan/eks-karyawan

## 5. Rekomendasi

### Immediate (≤24 jam)

1. **Revoke token `zrDZaIVCNUKphfjKpTAHA6uoVmASv8JnA2AjLGm5` sekarang** — sebagai precaution.
2. **Identifikasi pemilik asal token** dari database, korelasikan dengan status kepegawaian.
3. **Audit semua aktivitas token tersebut** di lumen log untuk 30 hari ke belakang (`grep -F 'zrDZaIVCNUKphfjKpTAHA6uoVmASv8JnA2AjLGm5' ...`) — bukan hanya endpoint cache, tapi semua akses.
4. **Tinjau ACL `internal_references.caches.delete`** — siapa saja yang punya, dan apakah orang yang sudah keluar masih punya.

### Short term (≤1 minggu)

5. **Refactor `Controller@clearCaches`** — ganti dari `FLUSHDB` ke invalidasi key-spesifik. Jika tidak bisa, batasi rate-limit (mis. 1×/menit per token).
6. **Implementasi token attribution di log** — field `user.email` harus selalu mengandung email pemegang token, bukan `system`, sehingga setiap akses terlacak.
7. **Rotasi semua Bearer token operasional** yang berumur >90 hari. Berlakukan kebijakan expiry maksimal 90 hari.

### Medium term (≤1 bulan)

8. **Hapus PAT GitHub hard-coded** dari user-data ASG dan `deploy.sh` (`ghp_Z68zw...` yang ada di repo) — masalah keamanan sejenis di lapisan deploy.
9. **Refactor pola N+1 cache lookup** di SalesInvoices mapper — root cause kerentanan kinerja yang membuat insiden ini terjadi mudah meledak.
10. **Pisahkan Redis DB per aplikasi** dengan konvensi yang didokumentasikan (mis. API=db5, HR=db6, Billing=db7) untuk mencegah blast radius silang di masa depan.
