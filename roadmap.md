# Roadmap Porting `api_v2` (PHP/Lumen) → `api_v2_go` (Go)

Dokumen ini mencatat progres porting backend Zahir Online dari PHP/Lumen ke Go.
Status: ✅ selesai · 🟡 sebagian · ⬜ belum.

Terakhir diperbarui: 2026-06-03 (**PATCH `sales_invoices` ditambahkan** — `internal/modules/sales/patch.go` + `router.go` + `create.go`. Set method HTTP `sales_invoices` kini lengkap: GET/POST/PUT/**PATCH**/DELETE. **(1) `PATCH /api/v2/sales_invoices/{id}`** didelegasikan ke `update()` dengan label method `PATCH` — untuk sales_invoices PATCH praktis identik dengan PUT (legacy `Transactions::updateOrCreate(method=PATCH)`: unpost lama bila approved → tulis ulang anak dari payload → repost; frontend mengirim payload penuh, PATCH hanya tombol "posting" dari daftar). `update()` direfactor menerima parameter `method` (PUT/PATCH) untuk histories. **(2) `PATCH /api/v2/sales_invoices/{id}/credit_limit_extension`** (padanan `SalesInvoicesCreditLimitExtension`): validasi `days` integer ≥0 (default 0) & `reason` wajib, lalu `UPDATE sales SET credit_limit_ext_days, credit_limit_ext_date=today WHERE transaction_id=id`, tulis histories (`end_point=sales_invoices_credit_limit_extension`). Re-validasi `CreditLimitException` sengaja dilewati karena endpoint ini justru melonggarkan batas. **Diuji live** (tenant `pelarikalcer260603114724-pg`): create draft `submitted` 4×100.000 (jurnal 0) → PATCH `approved` payload penuh → status approved, jurnal CR Penjualan 400.000 / DR Piutang 400.000, subsider receivables 400.000, histories `POST`+`PATCH`; PATCH credit_limit_extension `days=30` → `sales.credit_limit_ext_days=30`, `credit_limit_ext_date=today`, histories `sales_invoices_credit_limit_extension`; validasi: tanpa `reason`→422, `days` negatif→422, id ngawur→404; DELETE → g_jurnal deleted, jurnal/subsider 0 (cleanup bersih). **Sebelumnya: Lot & serial number `sales_invoices` LENGKAP end-to-end + modul prasyarat `inventory_adjustments`** — `internal/modules/sales/{lotserial.go,adjustments.go,inventory.go,create_calc.go,create_resolve.go}` + `router.go`. **(1) `lotserial.go`** mem-port subsistem lot/serial (Lots/Serials): tulis `LOT_LEDGERS`/`SERIAL_LEDGERS` (in/out, key `item_id`=baris transaksi), validasi ketersediaan (OUT: serial via Σ ledger, lot via PRODUCT_LOTS; IN: serial belum-terpakai), `postLotMaster`/`postSerialMaster` (padanan `Lots::postLot`/`Serials::postSerial`, join INVLEDGR → update `PRODUCT_LOTS`/`PRODUCT_SERIALS`), `unpostLotSerial` (balik master + hapus ledger, konsisten in & out). **(2) `inventory_adjustments`** (`adjustments.go`, endpoint baru POST/PUT/DELETE, ref-key 6 `IJ`): header G_JURNAL + line `INVJUR`, posting tulis `INVLEDGR` (BELI=masuk/JUAL=keluar dgn COGS), update `IGUDANG`/`INVENTOR`, jurnal penyesuaian per-baris (DR Persediaan / CR akun penyesuaian), lot/serial IN, validasi stok negatif, unpost penuh. Cleanup uji bersih (0 sisa).<br>**Fase 2b — posting persediaan (inventory ledger) + HPP/COGS `sales_invoices`** di `internal/modules/sales/inventory.go`. Saat posting (`approved`/`finished`), tiap line produk `is_tracked_as_inventory` menulis baris `INVLEDGR` (keluar stok `JUAL=base_quantity`; retur→`BELI`), dengan COGS metode **rata-rata bergerak** (`Inventories::getCogs`: Σ(beli−jual)·cost s/d tanggal, fallback last-purchase bila `inventory.is_use_cogs_adj_for_negative_stock`). Update master stok `IGUDANG` (per gudang) & `INVENTOR` (qtyonhand/totalnilai/hargapokoksatuan). **Validasi stok negatif** (`NegativeStockException`): tolak 422 bila `inventory.is_allow_negative_stock=0`; bila on → set `cost=0`. **Jurnal HPP transaksi anak terpisah** (`g_jurnal.parent_id=faktur`, status approved): DR akun HPP / CR akun Persediaan sebesar Σ|base_qty·cogs|, tulis `JURNALVC`+`JURNAL`. Status `finished` kini diperlakukan sama dengan `approved` (memicu posting). **Unpost** (`unpostInventory`, dipakai DELETE & PUT-repost): hapus jurnal HPP anak + `INVLEDGR` faktur, lalu recompute `IGUDANG`/`INVENTOR`. Guard `guardPostable` lama dihapus. **Diuji live** (tenant `pelarikalcer260603114724-pg`, produk uji `INVTEST1` stok awal 100@5.000): jual 10@12.000 → INVLEDGR jual 10 cost 5.000, IGUDANG/INVENTOR 90/450.000/5.000, GL CR Penjualan 120k/DR Piutang 120k, jurnal HPP anak DR HPP 50.000/CR Persediaan 50.000; diterapkan di `resolveLines`: `base_quantity (JUMLAH) = mult × quantity`, `base_price (HARGA) = price_origin (HARGA_A) = unit_price / mult` (amount GL tetap `quantity × unit_price`, tak berubah). **`latestExchangeRate`** (padanan `CurrencyRates::getLatestExchangeRate` + `Transactions::setCurrency`): bila `exchange_rate` tak dikirim & currency ≠ base → ambil `nilai` terbaru dari `KURSITEM` per tanggal (`tanggal <= date ORDER BY tanggal DESC`), fallback 1; input eksplisit tetap menang. **Diuji live** (tenant `pelarikalcer260603114724-pg`): unit Dzn→Pcs ×12 (qty 2 × 120.000 → JUMLAH 24, HARGA 10.000, HARGAF1 120.000, nilaipenjualan GL 240.000); USD (kurs 16.000 sementara di KURSITEM) tanpa `exchange_rate` → `g_jurnal.exchange_rate=16.000`, sales base 1.600.000 / origin 100; `exchange_rate` eksplisit 15.000 meng-override lookup; cleanup bersih. **Ditunda:** multiplier untuk produk inventory tetap (dipakai INVJUR/COGS Fase 2b), revaluasi/realisasi selisih kurs.<br>**Subsidiary ledger AR `sales_invoices`** di `internal/modules/sales/post.go`. Saat posting (`approved`), tiap payment non-cash — termasuk baris penyeimbang `receivable` — kini menulis baris `SUBSIDER` (buku besar piutang per-faktur): `type='receivables'`, `parent_id=child_id=<id faktur>`, `contact_id`=customer, `currency_id`, `total`=amount_origin, `employee_id`=salesman pertama (opsional). Padanan blok subsider di `Transactions::setPaymentsJournal` + `processSubsidiaryLedgers`. **Unpost** (`unpostInvoice`) kini juga `DELETE FROM subsider WHERE child_id=<id>` (dipakai DELETE & PUT repost). **Diuji live** (tenant `pelarikalcer260603114724-pg`): kredit penuh 400.000 → 1 baris receivables 400.000 (self); parsial tunai 100.000 dari 300.000 → subsider 200.000 + jurnal seimbang (CR Penjualan 300k / DR Kas 100k / DR Piutang 200k); tunai penuh → 0 subsider; DELETE → subsider terhapus; cleanup bersih (0 sisa g_jurnal/subsider/jurnal). **Ditunda:** `updateSubsidiaryBalance` recompute `sales.saldo` (sudah benar dari draft write, di-skip), guard `is_transaction_has_been_paid` (menunggu `receivable_payments`), `parent_memo` (linked transaction).<br>**Fase 2a — posting jurnal (GL) `sales_invoices`**: Status `approved` (di POST/PUT) memicu posting: build jurnal double-entry (CR akun penjualan per line sebesar net pasca-diskon-item & pra-final-discount, CR akun pajak per item, DR akun diskon untuk final discount, CR/DR other costs, DR piutang/kas/bank untuk payments + baris penyeimbang `receivable`), validasi debit==kredit, tulis raw ke `JURNALVC` + agregasi per-akun ke `JURNAL` (GeneralLedgers), set `g_jurnal.status='approved'`. Akun default dari `KONFIGUR` by key (`receivable`/`bank`/`cash` per-currency via `data_id`, `sales.discount`, `sales.tax`). Baris penyeimbang piutang juga ditulis sebagai `transaction_items` type `payments`. **Diuji end-to-end**: approved diskon item 10%+final 5% → CR Penjualan 450.000 / DR Diskon 22.500 / DR Piutang 427.500 (seimbang); PUT approved→approved repost tanpa duplikat; DELETE unpost; pembayaran tunai penuh → DR Kas 300.000 (payment_status=paid). **Ditunda (Fase 2b):** inventory ledger `INVJUR`+COGS (posting ditolak bila line punya produk `is_tracked_as_inventory`), lots/serials, realisasi & revaluasi selisih kurs, akun rounding auto-balance, `is_create_discount_journal_on_line_item_discount`, cost distribution, status `finished`.
Sebelumnya: write-path `sales_invoices` draft **lengkap CRUD non-posting**: POST + **PUT update** + **DELETE (soft)** di `internal/modules/sales` (`create.go`/`create_calc.go`/`create_resolve.go`). `prepareInvoice` (resolusi master-data + hitung total ala `Transactions::setLineItems`/`setFinalDiscount`/`setOthers`/`setPayments`) & `writeChildren` dipakai bersama create/update. PUT = full-replace anak (UPDATE G_JURNAL + DELETE+INSERT SALES & seluruh koleksi), nomor lama dipertahankan kecuali diubah (cek unik). DELETE = soft delete `G_JURNAL.status='deleted'` (padanan `Transactions::deleteById` non-hard; read-path sudah filter `status!='deleted'`). HISTORIES mencatat POST/PUT/DELETE. **Diuji end-to-end** (tenant `pelarikalcer260603114724-pg`): create `SJ000001` (100.000) & `SJ000002` (diskon item 10%+final 5% → 446.500); siklus create→PUT (qty 5×50.000 −10% = 225.000, anak ter-replace count=1, bukan duplikat)→DELETE (404 setelah hapus)→audit POST/PUT/DELETE terverifikasi di DB; guard 422/404 (status approved/finished, line_items kosong, customer salah, id invalid, not found). **Ditunda (Fase 2):** posting jurnal & inventory (status approved/finished) + unpost saat update/delete approved, INVJUR, unit-conversion multiplier, prepayment allocation, auto-balancing AR payment row, approval flow, digital payment, lots/serials, multi-currency exchange-rate lookup, import SI dari DO.
